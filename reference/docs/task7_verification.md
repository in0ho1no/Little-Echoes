## 設計上の決定

- **0x02待ちと考え中の分離**: SPEC §3.5・§5に従い、0x01の物理送信成功後から3秒間は
  `kWaitingAccept`（待機LED）、0x02受信後は`kThinking`（回転スピナー）とする。0x01の
  エンコード・USB送信時間は3秒の受理待ちへ含めない。0x03は0x02受信後かつ35秒期限内だけ受理する。

- **録音の送信/破棄の正本**: 長押し/単押しの境界（1.5秒）は button_input のジェスチャと
  audio_record のサンプル数の両方で表現されるが（static_assertで同値を担保）、送信/破棄の判定は
  **録音サンプル数（`StopResult`）を正本** とし、`kLongPress` ジェスチャは録音停止の駆動には
  使わない。DMA取りこぼしで実時間1.5秒以上でもサンプルが足りない場合に、中身の薄い音声を
  誤送信しないため。

- **トリプルクリックの押下と録音開始の優先順位**: `main.cpp` の `loop()` はジェスチャを押下エッジ
  より先に処理する。トリプルクリックは3回目の押下で即時確定するため、その押下で録音を開始せずに
  強制リセットへ入る（task3申し送り）。

- **録音赤LEDは長押し確定（1.5秒）で点灯（2026-07-12・実機確認で判明した"押下時の一瞬赤"を修正）**:
  押下エッジで録音データの取り込みは即開始するが、赤LEDは `kRecordingLedDelayMs`＝
  `button_input::kLongPressMs`（1.5秒）保持してから `poll()` が点灯させる。1.5秒は「長押し確定＝
  会話開始＝録音が送信対象になる」境界（録音の kKept/kDiscarded 境界 `kMinRecordingSamples`=1.5秒
  とも一致）であり、赤＝「この録音は送信される」の確定表示になる。単押し/ダブル/トリプルの各押下は
  いずれも1.5秒未満なので、タップでの一瞬赤は構造的に発生しない。録音音声は押下時点から欠落なく
  取り込むため、遅らせるのは表示のみ（§5-2）。トレードオフとして押下〜1.5秒は待機LED（緑）のままだが、
  「赤くなったら録音確定＝離せば送信」という明快なフィードバックになる。値は定数化済みで調整可能。
  （当初は250msの暫定しきい値だったが、長押し境界と揃える方が原則的なため1.5秒へ変更）

### 段階2: PC側ダミースクリプトが必要

ダミースクリプト **`scripts/task7_fw_probe.py`**（作成済み）で確認する。OpenAI API 非依存
（課金なし）。READY待ち→（マイコンの `0x01` を受けたら）WAV保存→`0x02` 返送→`0x03`（既知PCM）
送信、を行う。ポートは **既定でDTR/RTSをアサートして開く**（`ARDUINO_USB_MODE=0`/USB-OTG CDC は
DTRアサート時のみマイコンが送信するため。実PC側と同じ挙動。MODE=1相手や非アサート開きは
`--no-assert-dtr`）。

```powershell
uv run python scripts/task7_fw_probe.py                    # 自動検出・normal（0x02→0x03再生）
uv run python scripts/task7_fw_probe.py --mode no-accept   # 0x02を返さない（3秒タイムアウト確認）
uv run python scripts/task7_fw_probe.py --mode accept-only # 0x02のみ（35秒タイムアウト確認）
uv run python scripts/task7_fw_probe.py --mode error       # 0x02→0x04（エラー確認）
uv run python scripts/task7_fw_probe.py --effect 1 --reply scripts/pre_rec_thanks.wav  # お祝い演出
```

大容量0x03受信の不具合と解決（2026-07-12、解決済み）:

- 症状: `0x02` は届く（考え中スピナーに入る）のに大きい `0x03`（79KB、さらに40KBでも）が届かず、
  5秒BODYタイムアウトで破棄→35秒応答タイムアウト→エラー点滅。音は鳴らず。
- 切り分け: マイコン→PCの253KB(0x01)は成功、PC側は79KBを262msで送信完了、起動トーンで
  スピーカーは正常と確認。にもかかわらず40KBすら5秒で受信できない＝**`ARDUINO_USB_MODE=1`
  （内蔵USB HW CDC/JTAG=usb_serial_jtag）のhost→deviceバルク受信が遅く/取りこぼす既知の特性**が原因。
  RXバッファ拡張(32KB)・LED停止・ペーシング送信のいずれも効かなかった。
- **解決: `platformio.ini` を `ARDUINO_USB_MODE=0`（TinyUSB USB-OTG CDC）へ切替（2026-07-12）**。
  OTGはバルクエンドポイント＋フロー制御で取りこぼしなく高速受信でき、79KBが短時間で受信・再生でき、
  音声も再生されて待機へ復帰した（実機確認済み）。
  - 付随変更: プローブは**既定でDTRアサート**に変更（MODE=0はDTRアサート時のみマイコンが送信）。
  - 最終的にコアの`USBCDC`（`Serial`）を迂回する独自TinyUSB CDCへ移行したため、
    `Serial.setRxBufferSize()`は使用していない。
  - LED停止（`isReceivingBody`）: OTG化後、一度は「フロー制御で不要」として撤去したが、大容量
    （約1.15MB）が依然届かない件の対策として**再導入**（2026-07-12）。大BODY受信中は`main.cpp`が
    NeoPixel`show()`をスキップしCPUを受信へ集中させる（考え中スピナーは受信中一時停止する）。
    その後の切り分けで大容量失敗の真因は純粋なスループット不足（下記追記・
    `docs/issue_usb_throughput.md`）と確定し独自CDCで解決したが、LED停止は受信優先の
    安全弁として残置している（現行では停止は大BODY受信中の1〜3秒程度）。
- 書き込み: MODE=0では自動リセットが効かない場合があり、BOOT長押し＋リセットのダウンロードモードで
  書き込む（task0参照）。書き込み後はリセットで起動。列挙PIDが変わりCOM番号が変わり得るが、
  `port_discovery`はVID 0x303Aで判定するため自動検出は維持。
- 段階2の確認状況（2026-07-12）: お祝いエフェクト（`--effect 1`）・3秒/35秒タイムアウト
  （`no-accept`/`accept-only`）・エラー（`0x04`）は**成功**。約79KB〜187KBの応答再生も成功。

> **2026-07-12追記**: 大容量失敗の根本原因は「PC→マイコンUSB転送が約44KB/s（<音声レート48KB/s）で
> 頭打ち」であり、Arduinoコア2.0.17の`USBCDC`受信実装（per-byteキュー）と確定した。以下の
> タイムアウト調整では解決しない。**課題整理と対応方針は `docs/issue_usb_throughput.md` を参照**。

大容量応答（約1.15MB=24秒クリップ）の一時対策と最終解決（2026-07-12）:

- 症状: 187KBは成功するが、`--reply scripts/pre_rec_test24sec.wav`（約1.15MB）は届かず再生されない。
  トリプルクリック確認（⑥）も大容量再生が前提のため未確認。
- 原因: OTGでスループットは大幅改善したが、大容量では **①マイコンのBODY 5秒タイムアウト**と
  **②プローブの書き込みタイムアウト5秒**の両方に抵触していた（1.15MBの転送が5秒を超える）。
- 一時対策: ①`packet.h` の `kBodyTimeoutMs` を5s→20s、②プローブの`write_timeout`を
  5s→60s（`WRITE_TIMEOUT_SEC_LARGE`）へ変更した。
- 最終解決: 独自TinyUSB CDCで1.15MBを2.25秒（約512KB/s）へ改善し、BODYタイムアウトは
  PC側と同じ**5秒の無進捗判定**へ改訂した（SPEC §3.2 v1.3.2）。20秒設定は現行ではない。

起動音（再起動の目安）: `main.cpp` の setup() で控えめな音量の起動音を鳴らす
（`kBootToneVolume=50`/`kBootToneFreqHz=880`/`kBootToneMs=1000`＝約1秒）。再生用音量とは別で、
鳴らし終えたら再生用音量へ戻す。M5.Speakerの実音出力が生きていることの確認も兼ねる。

音量・ゲイン調整と音質切り分け（2026-07-12）:

- 再生音量 `kSpeakerVolume`: 200（曇って小さい）→255（音割れ）→**220**（調整中。1定数で変更可）。
- 録音ゲイン `kMicMagnification`=32: 実機録音WAVで「それなりに撮れている」ことを確認済み。
- 24秒クリップの「ひどい音質」は**元データ（`pre_rec_test24sec.wav`）自体の品質不良**が主因と

- クリーンな長尺評価用に `scripts/make_test_tone.py` を追加（純音メロディ20秒を合成。
  `uv run python scripts/make_test_tone.py` → `scripts/manual_check_output/test_tone_20s.wav`）。

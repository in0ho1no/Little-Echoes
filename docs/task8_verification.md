    25秒上限（35秒に対し10秒マージン）
  - 一過性事象の記録（2026-07-12）: 項目19の初回試行で一度だけ「0x02待ちの緑→3秒
    タイムアウトの赤点滅」が発生（マイコンリセット→main.py再起動直後の試行）。当時の
    `pipeline.py`は0x01受信をログしておらず、0x01がPCに届かなかったのか0x02がマイコンに

    `Received recording`ログの有無で経路を特定できる**（出ない=マイコン→PC、
    出る=PC→マイコンの問題）
  - 2回目の発生とFW診断機構の追加（2026-07-13）: 1ターン目でも再発し、このときのPC側
    ログは完全に正常（0x01受信263KB→0x02即時送信→Realtime 3.41秒→0x03送信0.81秒完了）
    だった。0x03の送信が全速で完了している＝マイコンは受信・ドレインできている＝
    **0x02（7バイト）だけがマイコンで処理されなかった**ことを強く示す。FWのRX経路
    （SPSCリング・TinyUSBコールバック・状態機械の受理条件）を机上レビューしたが
    決定的欠陥は未発見。そこで**FWに診断パケット（0x7F、SPEC外・デバッグ用）を追加**し、
    エラー遷移の全経路（3秒/35秒タイムアウト・0x01送信失敗・録音/再生失敗・状態外での
    0x02/0x03受信・RXリング読み捨て）で理由コード＋詳細をPCへ通知するようにした。
    `pipeline.py`が`Firmware diagnostic: reason=0x.. (...)`として警告ログに解読して出す。
    **確認には新FWの書き込みが必要**（`docs/how_to_setup_fw.md`。以降のtask8確認は
    このFWで行う）。次に再現したときのログの見方:
    - `reason=0x06`（accept timeout）＝0x02がマイコンに届いていない/失われた（本命の確定）
    - `reason=0x0A`（0x02 ignored, detail=状態番号）＝0x02は届いたが状態外で読み捨てた
    - `reason=0x04/0x05`（playback error, detail=PlaybackError番号）＝再生開始/継続の失敗
    - `reason=0x09`（0x03 ignored）＝エラー後に遅れて届いた0x03の読み捨て（随伴情報）
    - `reason=0x08`（rx ring dropped）＝受信リング満杯での読み捨てが実際に起きている
  - 3回目の発生と送信完了判定の強化（2026-07-13）: 診断入りFWで再発（緑のまま数秒→赤点滅）
    したが、**失敗ターンはPC側ログに一切痕跡を残さなかった**（0x01受信なし・診断なし。
    最後の正常応答から15秒でアイドル破棄が出ているのが不在の証拠）。今回は前回と逆方向で、
    **マイコン→PCの0x01自体がPCへ届いていない**。診断パケット（同方向）も届かないことと
    合わせて、「USB IN方向の転送が一時停止し、`writeAll`がFIFO受理時点で成功を返すため
    末尾がFIFOに残ったまま送信成功扱いになる」形の消失を想定原因とした。対策2点:
    ①PC側パーサの無進捗タイムアウト破棄に警告ログを追加
    （`Discarding stalled partial packet: cmd=.. expected_body=.. buffered=..`。
    「部分的に届いて止まった」ことが初めて見えるようになる）
    ②FWの`writeAll`をTX FIFO完全排出まで待って成功と判定するよう変更（末尾スタックを
    「送信失敗→即赤点滅＋診断」へ転換）。**FWの再書き込みが必要**。
    次回再現時の見方: PCに`Discarding stalled partial packet: cmd=0x01`が出れば
    「0x01が途中まで届いて停止」が確定。何も出ず即赤点滅なら送信自体が失敗
    （復旧後に`reason=0x01`の診断が遅れて届き得る）。
    それでも再現し続ける場合の追加実験: Windowsの「USBのセレクティブサスペンド」を無効化
    （電源オプション→USB設定）、または別のUSBポート/ケーブルで切り分ける。
  - **原因確定（2026-07-13）**: 上記①の警告が再現を捉えた。
    `Discarding stalled partial packet: cmd=0x01 expected_body=326656 buffered=326540`——
    0x01の**末尾116バイトだけ**が5秒以上届かない。すなわち**マイコン→PC（USB IN方向）の
    転送がパケット末尾で一時停止する**のが root cause。これで3回の失敗すべてが統一的に
    説明できる: 2回目（0x01は完着し0x02消失に見えた件）は「末尾が3〜4秒だけ停止→
    マイコンは3秒タイムアウトで赤点滅→直後に末尾が流れてPCは完全なパケットとして受信
    （Received recordingのログ時刻＝停止解消時刻）→もう待機に戻ったマイコンが0x02/0x03を
    仕様どおり読み捨て」。0x02のRX消失は最初から存在しなかった。
    機構の想定: この世代のTinyUSB（arduino-esp32 2.0.17同梱）の`write_flush`と
    転送完了処理の競合により、最終チャンクの送出予約が失われFIFOに残留する。
    旧`writeAll`は毎チャンクflushしており競合の窓を大量に踏んでいた。
  - **対策（2026-07-13、FW再書き込み必要）**: `writeAll`を3点変更。
    ①毎チャンクflushをやめ、flushは「FIFO満杯時」と「全量書き込み後」に限定（競合機会の削減）
    ②排出完了待ちループ内でflushを繰り返し再要求（送出予約の取り逃しからの自動回復）
    ③送信失敗時は`tud_cdc_n_write_clear`でFIFO残骸を破棄（次パケットへのごみ混入防止）。
    `pio run`ビルド成功・native 157件パス。**この対策後に再現しなくなることの確認が
    task8残項目の前提**（十数往復して停止・赤点滅・stalled partial packetログが出ないこと）
  - 対策v1では不十分＋発生条件の把握（2026-07-13）: flush削減＋再flush後も再発
    （`expected_body=626688 buffered=625932`＝末尾756バイト欠け）。再flushで回復しない＝
    詰まりはCDC flush層より深い。なお「長い発話ほど失敗しやすい」のは仕様violationではなく
    確率の問題（転送が長いほど競合を踏む機会が比例増。今夜の失敗2件は最長の2発話
    6.8秒/13.1秒。仕様上限30秒・1.44MBは段階2プローブで多数回成功しており、長さ自体は合法）
  - **対策v2（2026-07-13、FW再書き込み必要）**: TX送信をINエンドポイントの転送完了で
    直列化。TinyUSBの既知問題（[tinyusb #507](https://github.com/hathach/tinyusb/issues/507)系:
    アプリタスクのflushとUSBタスクの転送完了時auto-flushの競合で送出予約が失われる）を
    構造的に排除するため、`writeAll`を「`usbd_edpt_busy`が偽（転送完了）かつFIFO空を待つ→

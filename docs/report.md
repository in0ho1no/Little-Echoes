# 音声対話型スタンドアロンガジェット プロトタイプ開発報告書

本報告書およびソースコードの9割以上はAI製であることを念頭におくこと。  

## エグゼクティブ・サマリー

本プロジェクトは、音声入力、クラウド上の音声対話処理、音声再生および視覚フィードバックを一体化したスタンドアロン型ガジェットの実用プロトタイプを、土日の2日間で完成させることを目的として実施したものである。

総開発時間23.5時間（土曜9.5時間、日曜14時間。専用ケースの設計・製作は除く）で、M5Stack Atom EchoS3Rと37灯のUnit HEXを中核とする実機、Windows PC上の制御アプリケーション、OpenAI Realtime APIとの連携、堅牢なUSBシリアル通信、状態連動LED演出、物理キャンセル機構、および3Dプリント製専用筐体までを完成させた。

短期間での確実な完遂を最優先し、通信経路はWi-FiではなくUSB CDCによる有線直結、音声対話モデルは音声認識・応答生成・音声合成を一体で処理する `gpt-realtime-mini` を採用した。音声フォーマットを全経路で24 kHz、16 bit、モノラルPCMに統一し、変換処理を排除した結果、旧分離構成で14.34秒を要したクラウド処理に対し、実機E2E確認では新規セッション確立を含むRealtime応答約4.42秒を記録した。

PC側およびマイコン側の自動テストは合計298件を整備し、すべて合格した。加えて、実機による接続、音声対話、会話継続、定型応答・お祝い演出、タイムアウト、最大サイズ転送および再生中キャンセル等の主要経路を確認した。本成果により、デモ用途に留まらず、後続開発者が仕様に基づいて同等機を再構築し、Wi-Fi通信やストリーミング再生へ段階的に拡張できる技術基盤を確立した。

## システムアーキテクチャおよびハードウェア構成

### 全体構成

本システムは、マイコン、PC制御アプリケーション、OpenAI Realtime APIの3層で構成する。

1. マイコンが物理ボタン操作を受け、内蔵マイクから音声を24 kHz PCMで録音する。
2. 録音データをUSB CDCの独自バイナリプロトコルでPCへ転送する。
3. PCが `gpt-realtime-mini` のspeech-to-speechセッションへ音声を投入し、応答音声を取得する。
4. PCが応答PCMと演出IDをマイコンへ返送する。
5. マイコンが音声を再生し、Unit HEXで状態に応じたLED演出を表示する。

PC側は通信層、パケット層、会話パイプラインおよびRealtime APIクライアントを分離している。マイコン側はボタン入力、録音、再生、LED、シリアルリンク、パケットパーサおよび状態機械を独立モジュールとして構成している。この分割により、ハードウェア制御と対話ロジックの変更影響を局所化した。

### 採用ハードウェアと役割分担

| 構成要素 | 主要仕様 | 役割 |
| --- | --- | --- |
| M5Stack Atom EchoS3R（VoiceS3R） | ESP32-S3-PICO-1-N8R8、Flash 8 MB、PSRAM 8 MB、ES8311、NS4150B、MEMSマイク、物理ボタン | 録音、音声再生、USB CDC通信、ボタン入力、状態制御 |
| M5Stack Unit HEX | SK6812 RGB LED 37灯、Grove接続、データ線G2 | 状態・進捗・エラー・キャンセル受理の視覚表示 |
| Windows 11ノートPC | Python制御アプリケーション、USB Type-C接続 | シリアル制御、Realtime API接続、会話サイクル管理、デバッグ記録 |

Atom EchoS3Rは、マイク、スピーカー、アンプ、物理ボタン、USBおよび十分なPSRAMを小型筐体に集約できるため採用した。最大30秒の24 kHz PCMは約1.44 MBであり、8 MBのPSRAMを用いて録音・応答バッファを安全に保持できる。Unit HEXは37灯を用いた方向性のあるアニメーションが可能であり、音声のみでは伝えにくい「録音中」「処理中」「再生中」「異常」「キャンセル受理」を直感的に提示する。

LEDは電圧降下を防ぐため輝度を制限し、全37灯の白色フル点灯を避ける。NeoPixel更新にはESP32-S3のRMTペリフェラルを使用し、I2S DMAによる音声処理と並行動作させる。

### 3Dプリント製専用筐体

専用筐体は、Atom EchoS3RとUnit HEXを一体化した傾斜コンソール形状とした。単なる固定具ではなく、操作性、配線処理および視認性を同時に改善するユーザーインターフェースの一部として設計した。

- ボタン押下時に背面へ指を回して支えられる形状とし、小型機器にありがちな押下時の滑りや転倒を抑制した。
- 20 cmのGroveケーブルを筐体内へ収納できる空間を設け、余長の露出、引っ掛かりおよび外観上のノイズを排除した。
- Unit HEX前面に乳白色A4用紙を挿入できるディフューザースリットを設け、37灯の直接光による眩しさと点光源感を低減した。
- LED面を傾斜させ、卓上使用時に自然な視線角度から状態を確認できるようにした。

これにより、電子部品を接続しただけの試験機ではなく、片手操作、配線隠蔽、減光および状態認知を備えた実用的なデモ機として完成させた。

## 技術選定における意思決定（Rationales）

### USBシリアル直結方式

本プロトタイプではWi-Fiを採用せず、ESP32-S3のネイティブUSB CDCによるPC直結方式を採用した。判断基準は、2日間という固定納期内で、通信障害の発生要因を最小化し、再現可能な動作を確実に得ることである。

Wi-Fi構成では、アクセスポイント、認証、DHCP、電波環境、ファイアウォール、接続再確立およびネットワーク固有の遅延変動が障害要因となる。USB直結ではこれらを排除でき、給電と通信を1本のケーブルに集約できる。また、USB CDCネイティブ構成はボーレート指定が名目値であり、実測約512 KB/sの転送性能を確保した。1.15 MBの応答音声は約2.25秒、実測307 KBの応答は約0.59秒で転送できた。

一方で、将来の無線化を妨げないよう、PC側のパケット送受信は `Transport` インターフェースへ抽象化した。したがって、対話パイプラインを変更せず、シリアル実装をTCPまたはWebSocket実装へ置換可能である。

### `gpt-realtime-mini` の採用

音声対話にはOpenAI Realtime APIの `gpt-realtime-mini` を採用した。選定理由は、音声認識、言語応答生成および音声合成を単一のspeech-to-speechセッションに統合し、API呼び出し段数、データ変換および中間待ち時間を削減できるためである。

推論過程に追加時間を要するReasoning系モデル（2.1-mini）や、STT、LLM、TTSを個別に直列実行する構成は、個々の品質を調整しやすい反面、複数回のAPI往復と中間形式の受け渡しにより遅延と障害点が増加する。本プロトタイプでは最高精度よりも、ユーザーが自然に待てる体感速度と期限内の完成確度を優先した。

旧分離構成の実測はSTT 9.33秒、LLM 3.59秒、TTS 1.42秒、合計14.34秒であった。Realtime構成では、4.33秒の入力音声に対し、新規セッション確立を含む応答取得が約4.42秒となり、クラウド処理待ちを約69%短縮した。さらに、入出力PCMをAPIネイティブ形式へ統一したことで、リサンプリング、エンコードおよび外部変換ツールを不要とした。

### 明示操作と予測可能性の優先

音声入力はサーバーVADを無効化し、物理ボタンによるpush-to-talk方式とした。環境音による誤起動や発話終了の誤判定を避け、「長押し中だけ録音する」という利用者が理解しやすい規則を優先した。

会話履歴は常駐接続や無限のスライディングウィンドウではなく、最大3往復を1セッションとする。成功3往復、応答後15秒のアイドル、トリプルクリック、APIエラーのいずれかでセッション全体を破棄する。この方式により、履歴がいつ失われるかを明確にし、長時間接続に伴う無言の切断や状態不整合を構造的に抑制した。

## コア技術仕様および通信プロトコル要約

### 音声仕様

録音・API入出力・再生の全経路を以下の形式へ統一した。

| 項目 | 仕様 |
| --- | --- |
| フォーマット | ヘッダーなしPCM RAW、リトルエンディアン |
| サンプリングレート | 24,000 Hz |
| ビット深度 | 16 bit signed integer |
| チャンネル | モノラル |
| 最大録音時間 | 30秒（約1.44 MB） |
| 最短有効録音時間 | 1.5秒 |
| 最大応答時間 | 30秒（約1.44 MB） |

Realtime APIのpcm16要件と同一形式であるため、マイコンから受け取った音声を無変換でAPIへ投入し、API出力を無変換でマイコンへ返送する。これにより、リサンプリングによる音質劣化、処理遅延およびffmpeg等の追加依存を排除した。

### パケット構造

USB CDC上では、任意のバイナリ音声を安全に搬送するため、同期マーカー付き可変長パケットを使用する。

| フィールド | サイズ | 内容 |
| --- | ---: | --- |
| SYNC | 2 byte | 固定値 `0xA5 0x5A` |
| CMD | 1 byte | コマンドID |
| SIZE | 4 byte | BODY長、ビッグエンディアン |
| BODY | 可変長 | PCMまたは制御データ |

主要コマンドは、録音データ `0x01`、処理受理 `0x02`、音声再生 `0x03`、エラー `0x04`、キャンセル `0x05`、起動完了 `0x06` である。`0x03` のBODY先頭1 byteは演出IDであり、通常応答と感謝時のお祝い演出を識別する。

パーサは常にSYNCを探索し、破損データを読み捨てて再同期する。BODY上限は2 MBとし、上限超過を拒否する。BODY受信中は固定総時間ではなく「新しいデータが5秒間到着しないこと」をタイムアウト条件とする。これにより、低速でも進捗している最大1.44 MBの正常パケットを誤破棄せず、停止した送信元からは有限時間で復帰できる。

### タイムアウトと障害復帰

- 録音送信後、マイコンはPCからの `0x02` を3秒待機する。未受信時は赤色高速点滅3回の後、待機状態へ復帰する。
- `0x02` 受信後、マイコンは `0x03` または `0x04` を35秒待機する。期限超過時は同様に待機状態へ復帰する。
- PC側のRealtime応答待ちは25秒とし、マイコン側期限まで10秒の送信余裕を確保する。
- PC側シリアル書き込みには5秒の有限タイムアウトを設け、切断や送信停止による無限ブロックを防止する。
- Realtime通信失敗時はセッション状態を推測して継続せず、セッションを破棄する。アプリケーション制御で新規セッションによる再試行を1回に限定し、SDKの暗黙的な自動再試行は無効化する。

### `asyncio` 多重待機による即時キャンセル

PC側は単一OSスレッドの `asyncio` イベントループで構成し、Realtime WebSocketイベントとシリアル入力を同時に待機する。音声生成中も `0x05` を継続監視し、受信時は `response.cancel`、応答結果の破棄、セッション切断を順に実施する。同時期に応答完了が到着した場合もキャンセルを優先し、キャンセル後の音声をマイコンへ送信しない。

この構成は、ワーカースレッド間の共有状態やロックを導入せず、API応答待ち中の物理操作を即時受理できる。通信、タイマーおよびキャンセルの順序をイベントループ上で一元管理するため、異常終了時のセッション漏えいとシリアルクローズ待ちも抑制できる。

### RMT駆動と音声・LEDの並行制御

Unit HEXはESP32-S3のRMTペリフェラルで駆動する。RMTへ波形出力を委譲することで、CPUの長時間割り込み禁止を避け、I2S DMAによる録音・再生とLED更新を同時に継続できる。

主な表示は、待機中の緑ホタル点滅、録音中の赤、処理中の円形スピナー、通常再生中の青色周期明滅、感謝応答時のレインボー、エラー時の赤色高速点滅3回、短すぎる録音の黄色点滅、キャンセル受理時のマゼンタ点滅2回である。処理中および再生中もアニメーションを維持することで、利用者に停止状態と誤認させず、クラウド処理とデータ転送の待ち時間を体感上短くする。

### トリプルクリックによる物理強制リセット

初回押下から1.2秒以内の3回押下をトリプルクリックとして検出し、録音、処理、再生、待機の全状態で最優先に受理する。受理時は再生の即時停止、録音・再生バッファのクリア、PCへの `0x05` 送信、マゼンタ短点滅2回、待機状態への復帰を実行する。

マゼンタは、通常再生の青、録音破棄の黄、エラーの赤と明確に区別できる。したがって、利用者は操作が受理されたことを即座に確認でき、通信またはクラウド処理の完了を待たずに確実な初期化が可能である。待機中にセッションのみが残っている場合も同操作で会話履歴を破棄する。

## コスト・パフォーマンス分析

### クラウドAPIコストの暫定試算

採用モデル `gpt-realtime-mini` の音声単価が参照時点のモデルページで明示されていないため、以下は比較対象である `gpt-realtime-2.1-mini` の音声単価を同等と仮定した暫定値である。確定的な予算値ではなく、実運用前に最新の公式価格とAPIレスポンスのusageを用いて再計算する必要がある。

試算条件は、1セッション最大3往復、ユーザー発話30秒×3回、AI応答10～30秒×3回、音声入力100 msを1 Audio token、音声出力50 msを1 Audio token、入力100万token当たり10米ドル、出力100万token当たり20米ドルである。

| AI応答時間 | 1セッション暫定コスト | 1米ドル＝150円換算 |
| ---: | ---: | ---: |
| 10秒 × 3回 | 約0.036米ドル | 約5.4円 |
| 15秒 × 3回 | 約0.045米ドル | 約6.8円 |

### 応答性能とUXによる遅延吸収

旧STT＋LLM＋TTS構成のクラウド処理14.34秒に対し、Realtime APIでは実測約4.42秒まで短縮した。さらに307 KBの応答転送は0.59秒で完了しており、処理開始から再生準備までを実用的な待ち時間へ収めた。

待ち時間を無表示にせず、`0x02` 受理直後からUnit HEXの円形スピナーを表示し、応答受信後は青色明滅へ切り替える。これにより、利用者は入力受理、処理継続、再生開始を連続的に認識できる。視覚演出は実処理時間そのものを変えないが、無反応に見える時間を排除し、故障・フリーズの誤認と再操作を抑える効果を持つ。

## 将来の拡張性（ロードマップへのブリッジ）

### PC側通信のWi-Fi化

PC側は `send_packet(cmd, body)` と `recv_packet()` を提供する `Transport` 抽象を介して通信する。現在の `SerialTransport` を `TcpTransport` またはWebSocket実装へ差し替えることで、Realtimeクライアント、会話サイクル、キャンセル制御およびLED演出指示を変更せずにWi-Fi対応できる。

SYNC、CMD、SIZE、BODYからなるパケット形式もTCP上で再利用可能である。無線化時には、再接続方針、認証、暗号化、デバイス探索、パケット再送および帯域変動を追加設計する必要があるが、変更範囲はTransport層と接続管理へ限定できる。

### マイコン側ストリーミング再生

マイコン側の再生系は、受信データをリングバッファへ書き込み、再生タスクがリングバッファから読み出してI2S DMAへ供給する構造である。現行版はパケット全体の受信完了後に再生を開始するが、将来は受信部をチャンク単位の逐次書き込みへ変更することで、パケット受信と音声再生を重ねられる。

Wi-Fi化により帯域が低下または変動する場合でも、リングバッファの水位管理、再生開始しきい値、アンダーラン処理を追加すれば、対話ロジックと音声出力部を維持したままストリーミング化できる。これにより、応答音声の全量到着を待たずに再生を開始し、体感遅延をさらに短縮できる。

### 次段階で優先すべき拡張

次段階では、現行の明示的で予測可能な操作体系を維持したまま、以下を優先することが妥当である。

1. TCPまたはWebSocket Transportの追加と、切断・再接続・認証の仕様化。
2. チャンク転送とリングバッファ水位制御によるストリーミング再生。
3. API usageの継続収集による実コストモデルの確立。
4. 実利用環境別のレイテンシ、音量、マイクゲインおよびLED輝度の調整。
5. 長時間連続運転、ネットワーク断、最大長音声、電源変動を含む耐久・異常系評価。

本プロトタイプは、期限内の完成確度を高めるため有線・push-to-talk・全量受信後再生へ範囲を限定した一方、通信抽象化、リングバッファ、明示的セッション寿命および堅牢なパケット境界を備えている。したがって、現行成果を破棄せず、無線化、低遅延化および製品化評価へ段階的に移行可能である。

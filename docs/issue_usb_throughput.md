# 課題: USB CDC host→マイコン転送が約44KB/sで頭打ち（音声レート48KB/s未満）

作成: 2026-07-12（task7 段階2 実機確認で顕在化）。
ステータス: **解決（2026-07-12）**。対策A（独自CDC）の実機測定で
**1.15MBの転送が26〜27秒→2.25秒（約512KB/s、11.6倍）**に改善し、判定基準（10秒未満）をクリア。
24秒クリップの受信・再生を実機確認済み。音声レート48KB/sに対し10倍以上の余裕があり、
最大30秒応答（1.44MB）も約3秒で転送できる。§4.2の試算見直しは不要。

付随事項の顛末（2026-07-12、すべて解決）:
- 再生音質: 主因は24秒テストデータ自体の品質不良。クリーン音源（`scripts/make_test_tone.py`）では
  20秒再生が途切れ・ノイズ・音割れなし（音量220）を実機確認。再生経路は健全。
- READY自動再送: 接続（DTR）エッジ検知＋**200ms遅延送信**で解決（検知直後の送信はホストの
  オープン処理中のパージに飲まれるため）。ポートオープン直後のREADY受信を実機確認。

タイムアウトの最終状態:
- `kBodyTimeoutMs=5000`へ復帰済み。ただし「BODY全体を5秒以内」ではなく、SPEC §3.2に従う
  **5秒の無進捗判定**としたため、転送に進捗がある限り大容量BODYを誤破棄しない。

対策Aの実装内容（`firmware/src/serial_link.cpp`）:

- `ARDUINO_USB_CDC_ON_BOOT=0` とし、コアの `USBCDC`（Serial）を一切リンクしない
  （USBCDC.oの `tud_cdc_*` コールバック強シンボルとの衝突回避。`platformio.ini` 参照）。
- CDCインターフェースを `tinyusb_enable_interface()` で自前登録（ディスクリプタはコアと同一内容）。
- `tud_cdc_rx_cb`（USBデバイスタスク文脈）でTinyUSBのFIFOから**一括読み**し、PSRAM上の
  1MB SPSCリングへ蓄積。per-byteキューを排除し、パケット到着直後にエンドポイントを再アーム。
- `pumpIncoming()` はリングの連続領域を直接パーサへ渡す。TXは `tud_cdc_n_write` 直接
  （進捗1秒停止で打ち切り）。
- 付随改善: ホストのポートオープン（DTR立ち上がり）検知でREADYを自動再送（§3.4の
  「起動済みでREADYを逃した」ケースの救済が自動化。基板リセット不要になる見込み）。
- M5Unifiedの `Serial.begin` は `cfg.serial_baudrate=0` で抑止（main.cpp）。

検証: nativeテスト全パス、`pio run -e atom_echos3r` ビルド成功（リンク衝突なし）。
実機プローブで1.15MBを2.25秒で転送し、判定基準（10秒未満）を達成済み。

## 1. 課題サマリ

PC→マイコン方向のUSB CDC転送スループットが実測 **約42〜44KB/s** しか出ない。
音声データレートは 24kHz×16bit×モノラル = **48KB/s** であり、**転送が再生より遅い**。

- 最大30秒応答（約1.44MB）の転送に約33秒かかる計算。24秒クリップ（1.15MB）の実測で26〜27秒。
- マイコン側のBODY受信タイムアウト（現20秒）を超過し、パケットが破棄されて再生できない。
- タイムアウトを延ばせば「破棄」は防げるが、**20秒の応答を聞くのに約23秒待つ**体感は残る。
  将来のストリーミング再生化でも 44 < 48 KB/s のため underrun し、根本解決にならない。

逆方向（マイコン→PC、録音0x01）は 253KB を体感即時で送れており問題ない。
短い応答（canned_thanks 79KB≈1.65s、pre_rec_thanks 184KB≈3.9s）は現状でも成功する。

## 2. 実測データ

プローブ `scripts/task7_fw_probe.py` のログより（`0x03送信中…`→`0x03送信完了` の壁時計時間。
pyserialのwrite()がマイコン側のNAKバックプレッシャーでブロックされるため、これが実効受信レートを表す）:

| 構成 | BODYサイズ | 転送時間 | 実効レート |
| :-- | :-- | :-- | :-- |
| `ARDUINO_USB_MODE=1`（HW CDC/JTAG） | 79KB | 5秒超（タイムアウト） | 約15〜16KB/s |
| `ARDUINO_USB_MODE=0`（USB-OTG）切替後 | 79KB / 184KB | 数秒（成功） | — |
| 同上 | 1,153,569B | 27.26s / 26.32s / 26.82s（3回） | **42〜44KB/s** |

3回の独立測定で再現。ホスト側は分割ペーシング送信（4KBごと5ms）でも同時間（=ホスト側律速ではない）。

## 3. 試行して効果がなかった対策（時系列）

| # | 対策 | 結果 |
| :-- | :-- | :-- |
| 1 | `Serial.setRxBufferSize(32KB)`（既定256B→32KB） | 効果なし |
| 2 | 大BODY受信中の NeoPixel show() 停止（CPUを受信へ集中） | 効果なし |
| 3 | ホスト側ペーシング送信（`--send-chunk 4096`） | 効果なし |
| 4 | DTR/RTSアサート・非アサートの両方 | 効果なし |
| 5 | `ARDUINO_USB_MODE=1`→`0`（USB-OTG CDCへ切替） | **16→44KB/sに改善**したが依然不足 |
| 6 | メインループに `delay(1)`（下位タスクへyield） | 効果なし |
| 7 | 読み取りチャンク 256B→1KB | 効果なし |

アプリ側（loopTask側）の改善が一切効かないことが、後述の根本原因（USBタスク側の実装）と整合する。

## 4. 根本原因（コア実ソースで確定）

環境: PlatformIO espressif32@6.13.0 → **arduino-esp32 2.0.17**
（`framework-arduinoespressif32 3.20017.241212`。公式PlatformIOはこの系列で更新終了）。

`%USERPROFILE%\.platformio\packages\framework-arduinoespressif32\cores\esp32\USBCDC.cpp`:

- **L274-289 `USBCDC::_onRX()`**: TinyUSBのCDC FIFOから読んだデータを
  **1バイトごとに `xQueueSend(rx_queue, buf+i, 10)`** でFreeRTOSキューへ投入し、
  さらにパケット毎に `arduino_usb_event_post(..., portMAX_DELAY)` を呼ぶ。
- **L344 `USBCDC::read()`**: 読み出し側も **1バイトごとに `xQueueReceive`**。
- `sdkconfig.h`: **`CONFIG_TINYUSB_CDC_RX_BUFSIZE = 64`** — CDC FIFOは64B（バルクパケット1個分）。

`_onRX()` はTinyUSBデバイスタスクのコールバック内で実行される。FIFOが64Bしかないため
1パケット受けるたびに `_onRX()` が完走（per-byteキュー64回＋イベント投稿）するまで
エンドポイントが再アームされず、その間ホストへ **NAK** を返し続ける。

実測 44KB/s ≒ 64B ÷ 1.45ms。per-byteキュー操作＋イベント投稿＋タスク往復で
1パケット約1.4msかかる計算と一致する。`setRxBufferSize` はこのバイト単位キューの
「長さ」を変えるだけで、per-byteコストは不変（対策1が無効だった理由）。
ボトルネックはUSBタスク側のため、loopTask側の対策2/6/7も無効。

つまり **ハードウェア（ESP32-S3のUSB-OTG、フルスピード12Mbps）の限界ではなく、
Arduinoコアの `USBCDC` クラスの受信実装が律速**している。素のTinyUSB
（FIFOから一括読み）であれば数百KB/sが一般に報告されている。

## 5. 対応方針の選択肢

### A. USBCDCを迂回する独自CDC実装（推奨・根本対策）

Arduinoコアの `esp32-hal-tinyusb.h` が公開する `tinyusb_enable_interface()` で
CDCインターフェースを自前登録し、`tud_cdc_rx_cb` で **FIFOから一括読み**して
自前リングバッファへ入れる（per-byteキューを完全排除）。TXも `tud_cdc_n_write` を直接使う。

- 前提: `ARDUINO_USB_CDC_ON_BOOT=0` にして `Serial`（USBCDCインスタンス）を一切参照しない
  （参照するとUSBCDC.oがリンクされ `tud_cdc_rx_cb` が二重定義になるため）。
  `serial_link.cpp` のARDUINO部を書き換え、`USB.begin()`＋自前CDCで置き換える。
- 期待効果: 受信コールバックがmemcpyだけになり、エンドポイント再アームがµsオーダーに。
  数百KB/s級（最低でも録音方向と同等）が見込める。**プロトコル・SPEC・PC側は無変更**。
- 規模: firmware `serial_link.cpp` に約150〜200行（ディスクリプタ登録・DTR/ラインステート
  処理・RX一括取り込み・TX書き込みループ）。
- リスク: 列挙・DTR・書き込みモード遷移まわりの実機確認が必要（実機テストは必須）。
  失敗しても `ARDUINO_USB_CDC_ON_BOOT=1` に戻すだけで現状へ復帰できる。

### D. 音声圧縮（IMA ADPCM 4:1）で必要レートを下げる（確実なフォールバック）

0x03 BODYのPCMをIMA ADPCM（4:1）にすると必要レートは12KB/s。現状の44KB/sでも3.7倍の
余裕があり、**算術的に確実**。24秒クリップは288KB→転送約6.5秒。

- 規模: PC側エンコーダ（純Python約60行・pytest可）＋firmwareデコーダ（約60行・nativeテスト可）。
- 影響: **プロトコル変更**（SPEC §3.3/§4 の0x03 BODYフォーマットにコーデックを追加）。
  PC側 `main.py`／定型音声アセットにも波及（task8前に両側を揃える必要）。
- 音質: 音声用途では実用十分（16bit→4bit適応差分。電話品質以上）。

### C. Arduinoコア3.x系へ移行（pioarduinoプラットフォーム）

コア3.x（IDF5系）での改善に期待する案。ただし **3.xでもUSBCDCのper-byteキュー実装が
修正されている確証は現時点でない**（要確認）。プラットフォーム乗り換え（RMTドライバ変更等）の
リグレッションリスクに対し利得が不確実なため、優先度低。

### B. 制限を受容して先へ進む（最終フォールバック）

応答を短く（実用上〜8秒 ≒ 384KB ≒ 転送9秒）制限し、タイムアウトを応答尺に合わせて調整。
SPECの最大30秒応答の試算見直しが必要。デモの体感（応答の待ち時間≒応答の長さと同等）は妥協。

## 6. 推奨と次アクション

**A（独自CDC）を先に実施** → 実機でスループット再測定 →
- 十分（≳100KB/s）なら解決。プロトコル無変更で最大30秒応答が数秒〜十数秒で届く。
- 不足なら **D（ADPCM）** へ。こちらは算術保証があるが両側変更のため、Aの結果を見てから。

測定方法は現行プローブのまま（`0x03送信中…`→`送信完了` の時間差）。
判定基準: 1.15MBの転送が **10秒未満**（≒115KB/s以上）なら30秒応答も25秒応答待ち枠に収まる。
